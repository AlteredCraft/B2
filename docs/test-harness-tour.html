B2 — touring the harness, and the system beneath it

B2 is a local-first Markdown knowledge vault with an AI layer that discovers typed, explained connections between notes. The code that exists today is the index engine: a SQLite store treated as a disposable projection of your Markdown plus a thin event log. This page walks the test suite that engine is built against — what each test pins, and the shape of the thing being pinned.

51 / 51 tests passing 12 test files · 1 shared fixture module crate b2-core v0.1.0 Rust 1.96.1 · edition 2021 step 0 → 5 + accept complete

Verified by cargo test on the toolchain above — every step of the build ladder is green. The harness is integration-first: most tests open a real SQLite database, ingest a real vault on disk, and assert on the resulting projection.

The system under test

One crate, b2-core, ~2,150 lines of source across fourteen modules. Its job: turn a folder of Markdown into a queryable index — notes, chunks (full-text + vector), and a typed edges graph — and keep that index a pure function of the files on disk plus an append-only log.

Markdown vault + .b2/log/events.jsonl → ingest / replay → b2.sqlite (disposable) → search · neighbors · suggest · accept

The governing equation the whole suite exists to protect: index = projection of (Markdown ∪ log). Delete b2.sqlite, rebuild it, and it must come back identical. Only two things are durable and un-derivable: the Markdown itself, and the event log.

The load-bearing bet

FTS5 (BM25 keyword) and sqlite-vec (brute-force vector KNN) live in one statically-linked connection — no runtime load_extension, single binary. If that coexistence failed, the single-store premise of the whole design would be wrong. So it is literally the first thing the harness proves (step 0).

The invariants the harness defends

Almost every test is an assertion about one of these five properties. They are the reason the suite is shaped the way it is.

InvariantWhat it means · where it's pinned
Lossless round-tripparse → serialize → parse is byte-identical: unknown YAML keys, comments, whitespace all survive. B2 only ever makes surgical edits. → roundtrip.rs
B2 never authors the bodyThe body is the human's document. B2's writes are surgical and all but one are in frontmatter: the b2id stamp, and an accepted edge appended to relations:. The lone body write is the mechanical wikilink-path rewrite on move — repairing a link the human already wrote. → stamp.rs, frontmatter.rs, accept.rs
Idempotent re-projectionRe-ingesting changes nothing; an incremental one-note re-index equals a full rebuild; no stale FTS/vector rows accumulate. → chunks.rs, graph.rs, embed.rs
Disposable indexDrop the DB, rebuild from Markdown, replay the log → the review queue (and accepted edges) come back exactly. → suggestions.rs, accept.rs
Inert until acceptedA generated suggestion lives only in the queue + log — not one byte of any note changes; accepting it then writes only to frontmatter relations:, never the body. → suggestions.rs, accept.rs

The golden-vault fixture

Most integration tests share one tiny, hand-authored vault: two notes with a known link structure. It is copied into a temp dir before every run (golden_vault_copy) so ingest — which may stamp a b2id — never mutates the committed fixtures. Two fixed b2ids let the tests assert exact ids.

concepts/memory.md  ·  01JMEM…000A
--- b2id: 01JMEM0000000000000000000A type: concept title: "Human memory" created: 2026-06-20 --- The brain encodes, stores, and retrieves information…
notes/spaced-repetition.md  ·  01JSRS…000B
--- b2id: 01JSRS0000000000000000000B type: concept title: "Spaced repetition" created: 2026-06-20 --- Spaced repetition exploits the [[concepts/memory|Human memory]] retrieval curve. ## Relations - elaborates [[concepts/memory|Human memory]] — applies the forgetting curve
This 2-note body is engineered to exercise exactly two edge kinds at once: the prose [[…]] is a bare references edge; the - elaborates [[…]] — … list line is a typed edge with an explanation. Both are human-authored in the body (origin=inline) — B2 reads them, never writes them. A B2-accepted edge would instead land in frontmatter relations: (origin=frontmatter), keeping the body pristine. So a single ingest yields a known-good graph the tests can match whole.

The deterministic seams

B2's design tenet "build for tomorrow's model" puts every non-deterministic part behind a trait, so the harness can swap in a deterministic double. This is what makes the AI-adjacent engine testable at all.

Embedder seam — embed::Embedder

Tests run against FakeEmbedder: blake3-hashes the text into a vector, so identical text → identical vector, deterministically. KNN is reproducible and a drop-&-rebuild yields byte-identical vectors. The real local model (EmbeddingGemma / ONNX) drops in behind this same trait with no schema or flow change.

Honest scope note the suite states out loud: with a fake embedder, vector ranking isn't semantic — step 5 proves the plumbing (fusion math + the join), not model quality. Model quality is a future real-embedder eval suite.

Event-log seam — event::EventSink

Three interchangeable sinks: JsonlSink (durable .b2/log/events.jsonl), NullSink (drops events, for step 1–3 paths), and a test-local CollectingSink that captures what was appended so a test can assert the exact event emitted.

Plus FixedId / FakeId — a deterministic id generator — so stamped b2ids and suggestion ids are assertable byte-for-byte instead of random ULIDs.

The build ladder — step 0 → 5

The harness mirrors the build order from the spec. Each rung adds a tier of the engine and ships its own tests that re-assert the locked invariants against the golden vault. Read top-to-bottom and you read the system being assembled.

Decision 1 (2026-06-30): B2 never authors the body. Accepting a suggestion appends its typed-link string to the source note's frontmatter relations: — the same <verb> [[path|title]] — why syntax a human writes in the body, just in metadata. The graph is then the union of body links (inline) ∪ frontmatter relations (frontmatter) ∪ the log (suggested).

Edges of the harness — what is not yet tested

An honest tour names its boundaries. The suite itself is candid about these; they're tracked in the planning backlog, not gaps that slipped through.

  • The move operation (b2 mv). The one body write B2 makes — rewriting an inbound [[oldpath]] → [[newpath]] when a target moves (user-stories Story 1) — isn't built yet. The b2id-keyed edges already make it a bounded, foreign-key-true operation; it just needs wiring.
  • Semantic quality. Everything vector-related runs on the deterministic FakeEmbedder, so the suite proves plumbing, not ranking quality. A real-embedder eval suite is a separate, later effort.
  • No CLI / public API surface yet. Tests call the crate's internal functions directly; the b2 add/search/link/suggest/neighbors/reindex command surface and the typed core API are still backlog.
  • The discovery pipeline. Candidate generation → typed/explained suggestions → review loop exists only as its substrate (the graph⨝vector join + the suggestion lifecycle), not as an orchestrated pipeline.
