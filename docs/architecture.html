B2 — how the system is built

B2 is a local-first Markdown knowledge vault with an AI layer that discovers typed, explained connections between notes. The code that exists today is the index engine — a SQLite store treated as a disposable projection of your Markdown plus a thin event log — its real local embedder, and the typed Vault API the b2 CLI drives. This page describes how those pieces are constructed: the layers, the data flows through them, and the invariants that hold them together. Every claim here is ground truth — checked against the test suite, which is the executable specification of how the engine actually behaves.

Verified by cargo test on the toolchain above. The suite is integration-first: most tests open a real SQLite database, ingest a real vault on disk, and assert on the resulting projection — so it describes the built system, not mocks of it.

The governing equation

One equation shapes every design decision in the engine. If you keep only one thing from this page, keep this:

index = projection of ( Markdown ∪ log )
Markdown vault + .b2/log/events.jsonl → ingest / replay → b2.sqlite (disposable) → neighbors · search · suggest · accept

Delete b2.sqlite, rebuild it from the Markdown, replay the log — and it must come back identical. Only two things are durable and un-derivable: the Markdown itself (the source of truth for knowledge) and the append-only event log (the source of truth for history and review state). The index is a pure, disposable cache of both. Everything downstream — how writes are constrained, why the graph is materialized, how acceptance works — is this equation defended.

Three storage tiers

The system splits state into three tiers with sharply different durability contracts. Getting this split right is what keeps the vault pristine and the index honestly disposable.

Tier 1 · Markdown

Source of truth for knowledge. Notes + accepted edges, plain .md on your disk, fully usable with no B2.

Stays pristine. The body is 100% the human's — B2 authors none of it. Its only writes are surgical (see the write discipline).

Tier 2 · Index b2.sqlite

Disposable cache. Search indexes, the keyed graph, and the live review queue — everything the product reads on hot paths.

Holds nothing that can't be reconstructed from tiers 1 & 3. Drop it any time.

Tier 3 · Log .b2/log/

Durable, append-only history. Every consequential operation, with verbose payloads (model id, confidence, evidence).

Lose it and you lose only history and rejection memory — never a committed connection.

A whole vault is one portable folder: the index and log live under <root>/.b2/, a dotfolder Obsidian and the vault scanner both ignore. Point B2 at a folder of Markdown and that is the entire setup.

The substrate bet

B2's central architectural wager: one embedded store for every queryable concern at once — full-text (FTS5), vectors (sqlite-vec), the typed graph, and the live review queue — so candidate generation can join all four in a single transaction. That only works if FTS5 and sqlite-vec can share one connection.

How coexistence is achieved — db::open

FTS5 is built into SQLite. sqlite-vec is registered as a static auto-extension (sqlite3_auto_extension, once per process) so every later connection exposes the vec0 virtual table with no runtime load_extension — the whole thing is one statically-linked binary. Every connection opens WAL + foreign_keys=ON; an idempotent migration seeds schema_version exactly once (IF NOT EXISTS + INSERT OR IGNORE, so reopen is a no-op).

Why it's the first thing built

If BM25 keyword ranking and brute-force vector KNN couldn't live in one connection, the single-store premise of the entire design would be false — and everything built on top would be wrong. So coexistence is proven before anything else exists. sqlite-vec in practice today is brute-force KNN (a full scan), which is single-digit milliseconds at personal-vault scale; ANN is deliberately not assumed.

Ground truth: substrate.rs proves exactly this bet — fts5_and_sqlite_vec_coexist_in_one_connection runs a BM25 query and a vector KNN against the same statically-linked connection, and pragmas_and_schema_version_persist_across_reopen confirms WAL, foreign keys, and the once-only migration hold across a close/reopen.

The schema — a derived projection

Every table is a projection of (Markdown ∪ log); nothing here is a source of truth. The schema is derived from the data model — it must satisfy the model, never the reverse.

Two schema-level details do real load-bearing work. A CHECK constraint couples origin and status — an authored (inline/frontmatter) edge is always active; a suggested edge is always suggested or rejected — so the database itself refuses an illegal review state. And chunks_vec is created outside the base migration, because its FLOAT[N] dimension is a DDL literal pinned to meta.embed_dim: the vector space can only be shaped once the embedder's dim is known (see the embedder seam).

FLOW ① Ingest — turning a vault into an index

crates/b2-core/src/ingest.rs

Ingest is the write path that realizes the governing equation. Its one subtlety: link resolution must not depend on file order — a note can link to one the walker hasn't reached yet. So ingest runs in two phases.

walk *.md (dotfolders skipped, sorted) → Phase 1 · notes + chunks + vectors → fills the resolver → Phase 2 · edges
  • Phase 1 projects every note's frontmatter (upsert_note) and chunks its body (replace_chunks), embedding each chunk into chunks_vec through the seam. This fills notes(b2id, path) — the path → b2id resolver — for the whole vault.
  • Phase 2 derives each note's edges by parsing its body + frontmatter relations: and resolving every [[path]] against the now-complete resolver. A link the walker hadn't reached in phase 1 still resolves.
  • If a note has no b2id, phase 1 stamps one — mint a ULID, write the single surgical line to the file, emit a b2id.stamped event. This is B2's one always-allowed write during ingest.

The incremental path, ingest_file, re-projects a single note against an already-built index — and by construction produces the same rows a full rebuild would for that note. Both replace rather than accumulate: replace_chunks deletes a note's old chunks (the FTS triggers emit a 'delete' sentinel; stale chunks_vec rows are cleared explicitly, since that virtual table has no trigger back to chunks), and replace_authored_edges deletes only the note's inline/frontmatter edges — never the log-derived suggested rows, so a re-parse can't clobber the review queue.

Ground truth: one_note_reindex_equals_full (graph.rs) asserts an incremental re-index of one note yields a byte-identical edge set to a full rebuild, and that a second full reindex is identical again (idempotent); reindexing_a_note_does_not_leave_stale_fts_rows (chunks.rs) proves the replace-not-accumulate discipline; ingests_golden_vault_and_resolves_b2id_path_both_ways (ingest_resolver.rs) confirms the two-phase resolver works in both directions.

MODULE Notes & the surgical-write discipline

crates/b2-core/src/note.rs

The vault must stay the human's. So a parsed note keeps the raw text verbatim and records only the byte spans of the frontmatter block; serialization returns those raw bytes. Queryable fields are read out by a read-only YAML parse and are never used to re-serialize. The result: parse → serialize → parse is byte-identical — unknown keys, comment tokens, odd whitespace, a missing final newline, all survive.

Against that lossless backdrop, B2 makes exactly two kinds of write, both surgical and both in frontmatter:

  • stamp_b2id — insert one b2id: line at the top of the frontmatter (or synthesize a minimal frontmatter block if there is none). A strict no-op if an id is already present: never re-stamp.
  • add_relation — append one YAML-quoted typed-link string to the frontmatter relations: list (creating the block, or the whole frontmatter, when absent). The item is quoted so [[, |, and : are always YAML-safe; it errors rather than risk corrupting a pre-existing flow-style relations:.
Ground truth: roundtrip.rs pins losslessness (round_trip_preserves_unknown_keys_comments_and_whitespace); stamp.rs pins the one-line, idempotent, never-re-stamp behavior; frontmatter.rs pins block creation, append, safe quoting, and that every other byte — including the whole body — is preserved.

MODULES Chunks, links & the typed graph

crates/b2-core/src/chunk.rs · link.rs · graph.rs

Three modules turn a note body into the two things the product queries: searchable chunks and a typed, traversable graph.

Chunking — chunk.rs

A minimal paragraph splitter: maximal runs of non-blank lines, one chunk each. char_start..char_end always addresses the exact body slice that produced the text, so chunks stay anchored for later highlight/explain. The chunks schema already carries token_count and heading_path for a future qmd-style ~900-token, overlap-aware chunker — swapping it in is a pure re-projection (drop & rebuild), no schema or invariant change.

Link classification — link.rs

Two body constructs, both ordinary Obsidian Markdown: a bare [[path|alias]] in prose ⇒ an untyped, directed references edge; a list item - <verb> [[path|alias]] — why ⇒ a typed edge with an explanation. A typed line consumes its wikilink, so it isn't also double-counted as a bare reference. The verb must be lowercase-kebab, which keeps prose like - See [[x]] from being misread as a type. One parser (parse_typed_spec) serves both a body typed line and a frontmatter relations: entry — one syntax, two homes.

The materialized graph — graph.rs

A note's outbound links are parseable from that one file on demand — which is exactly why the edges table is disposable. It is materialized anyway, because three things a runtime parse can't serve cheaply are the whole point of a graph:

  • Inversion (backlinks). "Who points at X" can't be read from X — only from every other note. The table makes it one indexed lookup, not an O(vault) scan.
  • Typed multi-hop traversal. "Notes within k hops of X" is one SQL traversal, not a scan per hop.
  • The discovery join & the suggestion queue. The semantic⨝graph join (below) and the review queue (whose suggested edges are deliberately never on disk) can only exist over a materialized structure.

neighbors reads outbound edges (labeled with the verb) then inbound edges (labeled with the verb's inverse), over status='active' only — no reciprocal edge is ever stored. reachable_within is an undirected BFS over active edges (a note related either way is reachable), the hop set the discovery join filters on.

Ground truth: links.rs (8 tests) pins every classification edge case — bare-vs-typed, the no-double-count rule, the lowercase-kebab guard, the tolerated tail verb. graph.rs's golden_graph_has_references_and_elaborates_inline_active matches the whole edge set as an exact ordered snapshot, and the neighbors_* tests confirm inbound labels are the passive inverse forms with no reciprocal row.

MODULE The relation vocabulary

crates/b2-core/src/relation.rs

A closed 10-verb core — what discovery emits and queries rely on — plus a tolerated tail kept verbatim. Expressiveness lives in the tail; reliability lives in the core. Each core verb ships a display-only inverse label and a symmetry flag; adding a verb here is the whole change (a relaxable policy, not a structural assumption).

The closed core
  • references → referenced-by
  • relates ↔ (symmetric)
  • elaborates → elaborated-by
  • supports → supported-by
  • refutes → refuted-by
  • contradicts ↔ (symmetric)
  • example-of → has-example
  • part-of → has-part
  • supersedes → superseded-by
  • derived-from → source-of
Core vs. tail

A core verb maps to its inverse for inbound display; symmetric verbs are their own inverse and traverse both ways. A tail verb (anything a human writes, like inspired-by) is stored verbatim, never dropped, and treated as opaque — no inverse label, no special traversal. A tail verb that proves common can graduate into the core in a later version; stored data never changes.

Ground truth: a_tail_verb_is_kept_verbatim (links.rs) proves an out-of-core verb survives intact, and the inbound-neighbor tests confirm core verbs render as their inverse labels while directed edges store exactly one row.

SEAM The embedder — a fake for tests, a real model for users

crates/b2-core/src/embed.rs (seam + fake) · crates/b2-embed/ (real)

Producing embeddings inside a single binary is the one genuinely hard part, and it is orthogonal to the store — so it lives behind a trait. The engine is built and tested against a deterministic fake; the real local model drops in through the same seam with no schema or flow change. This is the "build for tomorrow's model" tenet made mechanical.

The trait — embed::Embedder

Four methods: model_id, dim, embed (a document/passage), and embed_query (defaulting to symmetric). embed is fallible — a real model runs tensor math that can fail, and the index path must surface that, not panic. Retrieval is asymmetric-ready: models that prompt queries and documents differently override embed_query.

FakeEmbedder (the CI default): blake3-hashes text into a vector, so identical text → identical vector, deterministically. KNN is reproducible and drop-&-rebuild yields byte-identical vectors. It is not semantic — it stands in for the real model so the fast suite stays model-free.

The real model — b2-embed::LocalEmbedder

A candle-backed BERT sentence embedder (BAAI/bge-base-en-v1.5, 768-dim, ungated), pure-Rust inference compiled into the binary — no external ONNX runtime. CLS-pool + L2-normalize (so sqlite-vec's L2 distance ranks by cosine); an asymmetric query prefix. The dim is read authoritatively from the model's own config.json, so config can't lie about it.

Not bundled. b2 init downloads + verifies it into a shared XDG cache; LocalEmbedder::load fails fast if the files are absent. Source is configurable (mirror, alternate repo, or a local path for offline installs) via a global TOML.

The swap is safe by construction. meta records (embed_model_id, embed_dim); ensure_embedding_space shapes chunks_vec at that dim and, on a model/dim change, drops and recreates it empty so a full re-embed follows. Crucially, opening a vault never mutates the vector space — a mismatch is caught and fixed at reindex, and a search against a stale index fails fast with ModelMismatch rather than returning silently-wrong results.
Ground truth: embed.rs pins determinism, one vector per chunk, KNN self-retrieval at ~zero distance, byte-identical re-embeds, and — critically — changing_dim_recreates_the_vector_space_and_clears_vectors, so a swap can never leave stale vectors. The 4 config unit tests in b2-embed pin source resolution (default / mirror endpoint / alternate repo / path sanitizing).

FLOW ② Hybrid retrieval

crates/b2-core/src/search.rs

BM25 keyword search and brute-force vector KNN are retrieved in parallel and fused with Reciprocal Rank Fusion (Σ 1/(k + rank + 1), k=60, borrowed wholesale from qmd). Results resolve up from chunks to notes.

query → BM25 over chunks_fts ⊕ vector KNN over chunks_vec → RRF (k=60) → top chunks → notes
  • The keyword half sanitizes arbitrary natural-language text into a safe FTS5 expression — alphanumeric terms, each double-quoted so nothing is read as an FTS5 operator, OR-joined for recall. A query with no usable terms yields no keyword hits (the vector half still runs).
  • The graph-filtered variant is B2's reason to exist: "nearest chunks whose note is within k typed hops of note X" — the chunks_vec ⨝ chunks ⨝ edges discovery join that candidate generation will run on.
  • The Vault façade resolves hits to note level: it pulls a wider chunk pool than the limit, dedups to the highest-scoring chunk per note, and attaches a one-line snippet.

A reranker (a cross-encoder over the fused top-N) and query expansion are deliberately deferred behind clean seams — they change ordering, not the store or the candidate set.

Ground truth: search.rs (8 tests) locks the fusion constant at k=60, proves a doc present in both lists outranks single-list winners (rrf_ranks_a_doc_present_in_both_lists_above_single_list_winners), and pins the discovery join — graph_filtered_search_restricts_to_reachable_notes excludes a text-matching note that is disconnected from the anchor. Honest scope: under the fake embedder the vector half isn't semantic, so these prove the plumbing (fusion math + the join), not model quality — which is a separate real-embedder eval (cargo run -p b2-embed --example eval), never in CI.

FLOW ③ The suggestion lifecycle — inert until accepted

crates/b2-core/src/suggest.rs · frontmatter home in note.rs

A discovered connection is inert: generating one writes the log + the index queue, and not one byte of any note. The three-state lifecycle is where the "B2 never authors the body" decision becomes mechanical.

(none) agent proposes → suggested accept → active (origin=frontmatter)

…or reject → a tombstone that blocks the same (src, dst, type) from ever being re-proposed.

  • Generate. Guarded by edge_exists — a connection already present in any status (active, pending, rejected) is never re-proposed. Log first (durable), then project the queue row + provenance. The timestamp is passed in by the caller, keeping this a deterministic, testable function.
  • Accept. The bridge, and the subtle part: B2 builds the typed-link string from the target's current path + title, appends it to the source note's frontmatter relations: (Markdown first — never the body), logs suggestion.accepted, drops the suggested row, and re-ingests the file so the edge re-materializes as origin='frontmatter' / status='active'. Acceptance is a re-projection, not an in-place status flip — which is exactly what keeps the accepted edge traceable to a real line in the Markdown.
  • Reject. Log suggestion.rejected and flip the row to rejected — it stays as rejection memory.
Ground truth: suggestions.rs proves a generated suggestion appears in the queue with full decision fuel while both notes stay byte-for-byte identical, and that a rejection tombstone blocks re-proposal. accept.rs proves accept writes frontmatter (body byte-unchanged), activates the edge as origin=frontmatter, and logs it — and that accepting an unknown edge id is a clean no-op.

MODULES The event log & replay

crates/b2-core/src/event.rs · replay.rs

The pending queue and rejection tombstones are the one part of the index not derivable from Markdown — suggestions are never written to notes. Replaying the durable log reconstructs them, which is what makes index = projection of (Markdown ∪ log) literally true.

The sink — event::EventSink

A thin append / read_all trait so the JSONL backing (one event per line at .b2/log/events.jsonl, line order = sequence order) can later become an append-only SQLite log with no data-model change. Three sinks: JsonlSink (durable), NullSink (drops events), and a test-local CollectingSink that captures exactly what was appended. Events: b2id.stamped and suggestion.generated|accepted|rejected.

Replay — replay::replay_log

A pure function of the log. generated inserts the suggested edge + provenance; rejected tombstones it; accepted is a queue no-op delete — the active edge re-derives from Markdown via Flow ①, so it is never double-counted as both a queue row and an active edge. Only suggestion.* events touch queryable state; the rest are history. An INSERT OR IGNORE harmlessly absorbs a generated event whose edge was already materialized from an accepted frontmatter relation.

Ground truth: drop_and_replay_reproduces_the_queue (suggestions.rs) does the literal thing — rm b2.sqlite → rebuild from Markdown → replay the log → the queue returns exactly; replaying_an_accepted_suggestion_leaves_no_queue_row and accepted_edge_survives_drop_rebuild_and_replay (accept.rs) prove the generate+accept pair nets to a single active edge, no phantom queue row.

ADAPTER The typed Vault façade & the b2 CLI

crates/b2-core/src/vault.rs · crates/b2-cli/src/main.rs

Everything above exists as modules the integration tests call directly. The Vault façade is the single typed entry point the CLI (and any future GUI) is the sole client of — it owns the connection, the durable log, the injected embedder, and the id generator, and exposes only what today's commands need: open / reindex / neighbors / search (a note ref resolves by path or b2id).

  • The façade returns display-ready view structs (NeighborView, SearchResult, ReindexReport) and distinguishes a typed NoteNotFound from "found, no results" — so the CLI stays a dumb printer.
  • The b2 binary holds no engine logic: it parses args, picks the embedder (real LocalEmbedder by default; B2_EMBEDDER=fake for an offline/dev mode, and what the CLI tests run under), injects it through open_with_embedder, calls the façade, and prints — human, or --json for agents.
  • Commands that embed (reindex, search) fail fast with "run b2 init" if the model is absent; neighbors is a pure graph query and needs no model. Internal errors are translated to generic, actionable messages that never leak sqlite/io/serde internals (B2_DEBUG adds the detail).
Prove a behavior once at the façade, again at the command surface. A future GUI is a second adapter that inherits both layers. The CLI is honest: when the vector half isn't semantic (fake embedder) it says so on stderr, keeping --json stdout pure data.
Ground truth: vault.rs (8 tests) pins the façade — open creates .b2/, reindex is idempotent and reports counts, all ref forms (path, path-without-.md, b2id) resolve to the same set, search is note-level with a snippet. cli.rs (8 tests) drives the built binary and checks both human and --json output, the positional-vault form, clean non-zero exits on unknown notes, and that the not-yet-semantic caveat rides stderr.

The invariants that hold it together

Almost every design choice above serves one of these five properties. They are the reason the system is shaped the way it is — and each is pinned by the suite.

InvariantWhat it means · where it's defended
Lossless round-tripparse → serialize → parse is byte-identical: unknown YAML keys, comments, whitespace all survive. B2 only ever makes surgical edits. → roundtrip.rs
B2 never authors the bodyThe body is the human's document. B2's writes are surgical and in frontmatter: the b2id stamp, and an accepted edge appended to relations:. → stamp.rs, frontmatter.rs, accept.rs
Idempotent re-projectionRe-ingesting changes nothing; an incremental one-note re-index equals a full rebuild; no stale FTS/vector rows accumulate. → chunks.rs, graph.rs, embed.rs
Disposable indexDrop the DB, rebuild from Markdown, replay the log → the review queue and accepted edges come back exactly. → suggestions.rs, accept.rs
Inert until acceptedA generated suggestion lives only in the queue + log — not one byte of any note changes; accepting writes only frontmatter relations:. → suggestions.rs, accept.rs

The deterministic seams

The engine is AI-adjacent, yet its whole suite runs with no live model, no wall clock, and no random ids. That is deliberate: every non-deterministic part sits behind a trait so a deterministic double can be swapped in. This is what makes the system testable at all — and it is the same "build for tomorrow's model" seam a more capable model drops into later.

Embedder — embed::Embedder

FakeEmbedder hashes text into a reproducible vector; the real candle model swaps in behind the identical trait. Tests get deterministic KNN; users get semantics.

Event sink — event::EventSink

Durable JsonlSink, drop-everything NullSink, and a CollectingSink that lets a test assert the exact event emitted.

Id generator — id::IdGen

Production mints real ULIDs; FixedId stamps deterministic ids, so b2ids and suggestion ids are assertable byte-for-byte. Timestamps are passed in, never read from a clock.

Grounded in the tests

This page is architecture-first, but nothing on it is aspirational: the test suite is the executable specification of the engine, and every property described above is asserted somewhere in it. The suite is integration-first — it opens real databases and ingests a real vault — so it describes the system as built. Most tests share one tiny hand-authored fixture, the golden vault: two notes with a known link structure, copied into a temp dir before every run so ingest never mutates the committed fixtures, with two fixed b2ids so exact ids are assertable.

The golden vault · concepts/memory.md
--- b2id: 01JMEM0000000000000000000A type: concept title: "Human memory" created: 2026-06-20 --- The brain encodes, stores, and retrieves information…
…and notes/spaced-repetition.md
--- b2id: 01JSRS0000000000000000000B type: concept title: "Spaced repetition" created: 2026-06-20 --- Spaced repetition exploits the [[concepts/memory|Human memory]] retrieval curve. ## Relations - elaborates [[concepts/memory|Human memory]] — applies the forgetting curve

This 2-note body is engineered to exercise two edge kinds at once: the prose [[…]] is a bare references edge; the - elaborates [[…]] — … line is a typed edge with an explanation. Both are human-authored (origin=inline) — B2 reads them, never writes them. So a single ingest yields a known-good graph the tests can match whole.

Not yet built

An honest architecture names its boundaries. These are tracked in the planning backlog, not gaps that slipped through.

  • The move operation (b2 mv). The one body write B2 will make — rewriting an inbound [[oldpath]] → [[newpath]] when a target moves — isn't wired yet. The b2id-keyed edges already make it a bounded, foreign-key-true operation.
  • Semantic quality in CI. The engine suite runs on the deterministic fake, so it proves plumbing, not ranking quality. The real bge model is measured by a separate b2-embed eval, deliberately out of CI.
  • Only four commands so far. init / reindex / neighbors / search exist; add, mv, suggest / accept / reject, and explain are still backlog (their engine substrate is built and tested — only the command surface is missing).
  • The discovery pipeline. Candidate generation → typed/explained suggestions → review loop exists only as its substrate (the graph⨝vector join + the suggestion lifecycle), not yet as an orchestrated pipeline.
