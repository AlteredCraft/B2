B2 — how the system is built

B2 is a local-first Markdown vault with an AI layer that discovers typed, explained connections between notes. This repo is the index engine + CLI: a SQLite store treated as a disposable projection of your Markdown plus a thin event log, a real local embedder, a real Claude-backed relator, and the typed Vault API the b2 binary drives. This page traces the whole architecture through one real vault — the two-note fixture the test suite runs on — so every claim is concrete and checkable.

160 tests · model-free, deterministic 4 crates: b2-core · b2-embed · b2-relate · b2-cli Rust 1.96 · edition 2021 v1 kernel: reindex · search · neighbors · explain · add · mv · suggest · accept · reject

The suite is integration-first: most tests open a real SQLite database, ingest a real vault on disk, and assert on the resulting projection — so it describes the built system, not mocks of it. Two additional #[ignore] tests exercise the real model + real API and run out of CI.

The governing equation

One equation shapes every decision. Keep only this:

index = projection of ( Markdown ∪ log )
Markdown vault + .b2/log/events.jsonl → ingest / replay → b2.sqlite (disposable) → search · explain · suggest · accept

Delete b2.sqlite, rebuild it from the Markdown, replay the log — it must come back identical. Only two things are durable and un-derivable: the Markdown (source of truth for knowledge) and the append-only event log (source of truth for review state — pending suggestions and rejection tombstones). Everything downstream is this equation defended.

Tier 1 · Markdown

Notes + accepted edges, plain .md, fully usable in Obsidian with no B2. Stays pristine — B2's only writes are surgical and in frontmatter.

Tier 2 · b2.sqlite

FTS5 + sqlite-vec + the typed graph + the review queue. Disposable: holds nothing not reconstructible from tiers 1 & 3.

Tier 3 · .b2/log/

Append-only events.jsonl. Lose it and you lose only review history — never a committed connection.

A whole vault is one portable folder: index + log live under <root>/.b2/, a dotfolder Obsidian and the vault scanner both ignore. Point B2 at a folder of Markdown — that is the entire setup.

Four crates, cleanly layered

The engine owes nothing to the CLI above it or the two models beside it. Both AI-heavy crates sit behind traits defined in b2-core, so the engine and its whole fast suite never import a tensor or an HTTP client.

b2-core · the engine

Turns a folder of Markdown into a queryable index and keeps it a pure function of disk + log. Model-free (no candle, no network) so its suite is fast and deterministic. rusqlite (bundled SQLite + FTS5), sqlite-vec, blake3, ulid, yaml.

  • note · ingest · chunk · link · relation
  • db · graph · search · discover
  • embed · relate — the two seams + fakes
  • event · replay · suggest
  • add · mv · vault (the façade)
b2-cli · the adapter

The b2 binary. Holds no engine logic: parse args, pick + inject the embedder/relator, call the Vault façade, print (human, or --json for agents). Funnels every error through user_message so nothing internal leaks.

b2-embed · the real embedder

LocalEmbedder: candle-backed BERT, BAAI/bge-base-en-v1.5 (768-dim, ungated), pure-Rust inference. b2 init downloads + verifies it into a shared XDG cache. Implements embed::Embedder.

b2-relate · the real relator

ClaudeRelator: Claude-backed classification over raw HTTP (ureq, no tokio). Default claude-opus-4-8; key from ANTHROPIC_API_KEY. Implements relate::Relator. Backend is pluggable.

Ground truth: the layering is enforced by where deps live. b2-core's Cargo.toml has no candle and no HTTP client; the whole engine suite runs on the deterministic fakes. The real model is reached only by b2 init + the two out-of-CI evals.

FLOW ①  Ingest — vault becomes index

crates/b2-core/src/ingest.rs

b2 reindex is the write path that realizes the governing equation. Its one subtlety: a note can link to one the walker hasn't reached yet, so link resolution must not depend on file order. Ingest runs in two phases.

walk *.md (dotfolders skipped, sorted) → Phase 1 · notes + chunks + vectors → fills the resolver → Phase 2 · edges
  • Phase 1 projects each note's frontmatter, chunks its body (a minimal paragraph splitter, one chunk per maximal run of non-blank lines), and embeds each chunk through the seam. This fills notes(b2id, path) — the path ⇄ b2id resolver — for the whole vault. Our two notes yield one chunk each; memory.md's chunk lands in chunks_vec ready to be found.
  • Phase 2 derives edges by parsing each body + frontmatter relations: and resolving every [[path]] against the now-complete resolver. From spaced-repetition.md that's two directed edges to memory.md: a bare references and a typed elaborates. A typed line consumes its wikilink, so it isn't double-counted as a bare reference.
  • If a note lacks a b2id, phase 1 stamps one — mint a ULID, write a single surgical line, emit a b2id.stamped event. B2's one always-allowed write during ingest. (Both fixture notes are pre-stamped, so nothing is written here.)

Incremental by default, ≡ a full rebuild. A note whose body_hash is unchanged and whose chunks all still have vectors reuses them verbatim — only edited notes re-embed. Both paths replace rather than accumulate: re-projecting a note deletes its old chunks (FTS triggers fire, stale chunks_vec rows are cleared) and its inline/frontmatter edges — never the log-derived suggested rows, so a re-parse can't clobber the review queue. b2 reindex --dry-run previews the same decision, writing nothing.

Ground truth: graph.rs · one_note_reindex_equals_full — incremental re-index of one note yields a byte-identical edge set to a full rebuild, and a second reindex is identical again (idempotent). chunks.rs proves replace-not-accumulate; ingest_resolver.rs confirms the two-phase resolver works both directions; dry_run.rs proves the preview writes nothing. ↳ the indexing deep dive covers incremental skip, --force, batching.

The schema — a derived projection

Every table is a projection of (Markdown ∪ log); nothing here is a source of truth. The schema is derived from the data model — it must satisfy the model, never the reverse.

TableHoldsDerived from
notesb2id (PK), path (UNIQUE), type, title, created/updated, body_hash — the resolver is this tableMarkdown
chunksper-note paragraph chunks with char offsets + token_countMarkdown
chunks_ftsFTS5 BM25 index, kept in lockstep by triggersderived (FTS5)
chunks_vecvec0 KNN table, created at the embedder's dim — not in the base migrationderived (sqlite-vec)
edgestyped graph: src→dst, type, origin∈{inline,frontmatter,suggested}, status∈{active,suggested,rejected}, explanationMarkdown + log
edge_provenanceby / source / confidence / created / decided — a pending suggestion's decision fuellog
metaschema_version, embed_model_id, embed_dimbookkeeping

Two schema details do load-bearing work. A CHECK constraint couples origin and status — an authored edge is always active, a suggested edge is always suggested or rejected — so the database itself refuses an illegal review state. And chunks_vec is created outside the base migration because its FLOAT[N] dimension is a DDL literal pinned to meta.embed_dim: the vector space can only be shaped once the embedder's dim is known.

The substrate bet. One embedded store answers every query at once — FTS5, vectors, the typed graph, the review queue — so discovery can join all four in one transaction. That only works if FTS5 and sqlite-vec share one connection: sqlite-vec is registered as a static auto-extension (no runtime load_extension), so the whole thing is one statically-linked binary. substrate.rs proves a BM25 query and a vector KNN run against the same connection.

FLOW ②  Search — hybrid retrieval

crates/b2-core/src/search.rs

BM25 keyword search and brute-force vector KNN run in parallel and fuse with Reciprocal Rank Fusion (Σ 1/(k + rank + 1), k=60). Results resolve up from chunks to notes.

# keyword + semantic + graph, one command
$ b2 search "how does forgetting affect recall"
notes/spaced-repetition.md  Spaced repetition
    …exploits the [[concepts/memory|Human memory]] retrieval curve.
concepts/memory.md       Human memory
    The brain encodes, stores, and retrieves information…
query→ BM25 over chunks_fts⊕ vector KNN over chunks_vec→ RRF (k=60) → top chunks → notes
  • The keyword half sanitizes raw natural language into a safe FTS5 expression — each alphanumeric term double-quoted so nothing is read as an operator, OR-joined for recall. (Punctuation is FTS5 syntax and would otherwise crash the parse — a real bug the eval surfaced.)
  • A graph-filtered variant — "nearest chunks whose note is within k typed hops of note X" — is the chunks_vec ⨝ chunks ⨝ edges join. It searches within an already-related neighborhood (near ∩ connected). Discovery is its complement (near ∖ connected) — the two are opposites, and confusing them is the trap the engine turns on avoiding.
  • The façade resolves hits to note level: pulls a wide chunk pool, dedups to the best chunk per note, attaches a one-line snippet.
Ground truth: search.rs (8 tests) locks k=60, proves a doc in both lists outranks single-list winners, and pins the join — graph_filtered_search_restricts_to_reachable_notes excludes a text-matching note disconnected from the anchor. Honest scope: under the fake embedder the vector half isn't semantic, so these prove the plumbing, not model quality — that's the separate out-of-CI retrieval eval (cargo run -p b2-embed --example eval).

FLOW ③  Discover a connection — the reason B2 exists

crates/b2-core/src/discover.rs · relate.rs · crates/b2-relate/

Discovery proposes links you haven't made. It is a three-stage pipeline, now real and wired end to end: candidate generation (deterministic) → the relator (Claude) → the review queue (deterministic). b2 suggest runs the whole thing.

The one idea — a candidate is near ∖ connected

A candidate is a note semantically near the anchor but not already connected. Mechanism, per anchor:

anchor's stored chunk vectors→ KNN over chunks_vec→ max-sim per candidate note− reachable_within(anchor, 1)→ rank → top-N→ Relator
  • No re-embedding. Each anchor chunk's already-stored vector is the KNN query — discovery is passage↔passage, so the anchor is its indexed vectors, never an embed_query whose asymmetric prefix is the wrong side.
  • Max-sim to the note. Every other note scores by its single best chunk-pair similarity, carrying the winning chunk as evidence for the relator.
  • Subtract the graph — the only graph read. reachable_within(anchor, 1) (anchor + direct neighbors) is removed. Distance is exclusion-only: a 2-hop note (triadic closure — "you linked A–B and B–E but never A–E") survives, ranked purely by semantic score.

In our fixture the two notes are directly linked, so each is excluded as the other's candidate — a correctly empty result, which is exactly the assertion discover.rs pins.

The relator seam — relate::Relator

One method: relate(anchor, candidate) → Result<Option<Proposal>>, pairwise. Ok(None) is a first-class decline — generation over-produces, the relator prunes. A fired Proposal { edge_type, explanation, confidence } maps 1:1 onto generate_suggestion. The real ClaudeRelator lives in b2-relate; FakeRelator (content-addressed on the b2id pair, declines 1-in-4) drives CI with no LLM.

How the real relator stays honest

Structured output is forced tool use: one classify_relation tool whose input_schema pins relation to the closed core set via enum + tool_choice — the model returns a typed verdict, not free text. The pipeline still re-validates relation::is_core before persisting: the type gate is never trusted to the model. The key comes from ANTHROPIC_API_KEY and is checked at construction — a missing key fails fast, never a mid-run 401.

Ground truth: discover.rs (7) pins the complement — direct neighbor excluded, disconnected near note kept, 2-hop note survives, deterministic ranking. generate.rs (9) drives the full pipeline on stubs: fire-core / decline / non-core-dropped, determinism across rebuild, idempotent re-run, queue survives drop→rebuild→replay. b2-relate (11 model-free) pins the forced tool + verb-enum + response parsing; a #[ignore] live smoke test hits the real API out of CI. ↳ the discovery run deep dive walks a full b2 suggest.

FLOW ③ cont.  Accept or reject — inert until you say so

crates/b2-core/src/suggest.rs · frontmatter home in note.rs

A discovered connection is inert: generating one writes the log + the index queue, and not one byte of any note. Three states, where "B2 never authors the body" becomes mechanical.

(none)agent proposes → suggestedb2 accept → active (origin=frontmatter)

…or b2 reject → a tombstone that blocks the same (src, dst, type) from ever being re-proposed.

  • Generate. Guarded by edge_exists — a pair present in any status is never re-proposed (and, for the paid relator, skipped before the call). Log first, then project the queue row + provenance.
  • Accept. B2 builds the typed-link string from the target's current path + title, appends it to the source note's frontmatter relations: (Markdown-first — never the body), logs it, drops the suggested row, and re-ingests the file so the edge re-materializes as origin=frontmatter / status=active. Acceptance is a re-projection, not a status flip — which is what keeps the edge traceable to a real line in the Markdown.
  • Reject. Log it and flip the row to rejected — it stays as rejection memory.
what accept writes into the source note
---
b2id: 01JSRS…B
type: concept
title: "Spaced repetition"
relations:
  - "elaborates [[concepts/memory|Human memory]]"  # ← appended
---

Frontmatter only. The body is byte-unchanged. On the next reindex this line re-derives as an active edge — no phantom queue row.

Ground truth: suggestions.rs proves a generated suggestion appears in the queue with full decision fuel while both notes stay byte-for-byte identical, and a tombstone blocks re-proposal. accept.rs proves accept writes frontmatter (body byte-unchanged), activates the edge, logs it, and that an unknown id is a clean no-op.

The write discipline — losslessness & surgical edits

crates/b2-core/src/note.rs · add.rs · mv.rs

The vault stays the human's. A parsed note keeps its raw text verbatim and records only the byte spans of the frontmatter block; serialization returns those raw bytes. So parse → serialize → parse is byte-identical — unknown keys, comments, odd whitespace, a missing final newline all survive. Against that backdrop, B2 makes only four narrow writes:

Frontmatter, surgical
  • stamp_b2id — one b2id: line; strict no-op if present, never re-stamped.
  • add_relation — one YAML-quoted typed-link appended to relations: (the accept path).
Whole-file ops (CRUD)
  • b2 add — create a note (minimal frontmatter, no b2id; ingest stamps it on first sight).
  • b2 mv — move/rename + rewrite inbound [[oldpath]] link text.
b2 mv never breaks the graph. Edges key on b2id, and a move leaves the target's id untouched — so neighbors/backlinks show the same set before and after. Only the human convenience-copy [[oldpath|alias]] text is repaired, and only in the exact inbound files the materialized graph names (db::inbound_edge_targets — bounded, never an O(vault) scan). A byte-preserving rewrite swaps only the target token; a prefix-sharing [[foo-bar]] is never touched when moving foo.
Ground truth: roundtrip.rs pins losslessness; stamp.rs · frontmatter.rs pin the surgical writes; mv.rs (15 tests) pins graph-unchanged, byte-exact inbound diffs, prefix-safety, and clean errors on clobber/invalid dest; add.rs pins stamp-on-create and clobber refusal.

The two AI seams — a fake for CI, a real model for users

Every AI part sits behind a swappable trait defined in b2-core. The engine is built and tested against a deterministic fake; a real model drops in through the identical seam with no schema or flow change. This is the "build for tomorrow's model" tenet made mechanical — and it is why the whole suite runs with no live model, no wall clock, and no random ids.

Embedder — embed::Embedder

Fake: FakeEmbedder blake3-hashes text into a vector — identical text → identical vector, deterministically. KNN is reproducible; not semantic. Real: b2-embed::LocalEmbedder, candle BERT, CLS-pool + L2-normalize (so sqlite-vec's L2 ranks by cosine), asymmetric query prefix, dim read from the model's own config.json.

The swap is safe by construction. meta records (embed_model_id, embed_dim); on a model/dim change chunks_vec is dropped + recreated empty so a full re-embed follows. Opening a vault never mutates the vector space; a stale search fails fast with ModelMismatch rather than returning silently-wrong results.

Relator — relate::Relator

Fake: FakeRelator hashes the b2id pair into a reproducible verdict (a core verb, or a decline), so a discovery run is reproducible and never reads note text. Real: b2-relate::ClaudeRelator, forced tool use pinning the verb enum, default claude-opus-4-8 (config-overridable to claude-haiku-4-5 for cheap high-volume runs).

Injected as an argument to Vault::generate_suggestions(&dyn Relator, top_n) — a single consumer, so it stays off the façade surface while b2-core stays model-free. Both fakes are content-addressed, so drop→rebuild→replay is reproducible under either.

the relation vocabulary — closed core + tolerated tail · relation.rs
  • references → referenced-by
  • relates ↔ (symmetric)
  • elaborates → elaborated-by
  • supports → supported-by
  • refutes → refuted-by
  • contradicts ↔ (symmetric)
  • example-of → has-example
  • part-of → has-part
  • supersedes → superseded-by
  • derived-from → source-of

A closed 10-verb core is all discovery ever emits and all queries rely on; each ships a display-only inverse label and a symmetry flag (inbound neighbors render the inverse; only one directed row is ever stored). A tail verb a human writes (e.g. inspired-by) is kept verbatim, never dropped, treated as opaque. Expressiveness lives in the tail; reliability lives in the core.

The event log & replay — closing the equation

crates/b2-core/src/event.rs · replay.rs

The pending queue and rejection tombstones are the one part of the index not derivable from Markdown (suggestions are never written to notes). Replaying the durable log reconstructs them, which is what makes index = projection of (Markdown ∪ log) literally true.

The sink — event::EventSink

A thin append/read_all trait so the JSONL backing (one event per line, line order = sequence order) can later become an append-only SQLite log with no data-model change. Events: b2id.stamped and suggestion.generated|accepted|rejected. A b2 mv is not logged — it's fully reconstructible from Markdown (files at new paths, ids intact).

Replay — replay::replay_log

A pure function of the log. generated inserts the suggested edge + provenance; rejected tombstones it; accepted is a queue no-op delete — the active edge re-derives from Markdown via Flow ①, so it's never double-counted. Only suggestion.* events touch queryable state; the rest are history.

Ground truth: drop_and_replay_reproduces_the_queue does the literal thing — rm b2.sqlite → rebuild from Markdown → replay the log → the queue returns exactly; accepted_edge_survives_drop_rebuild_and_replay proves generate+accept nets to a single active edge with no phantom queue row.

The command surface — a dumb adapter over one typed façade

crates/b2-core/src/vault.rs · crates/b2-cli/src/main.rs

Every engine module is called directly only by the integration tests. The Vault façade is the single typed entry point the CLI (and any future GUI) is the sole client of — it owns the connection, the durable log, the injected embedder, and the id generator, and returns display-ready view structs (NeighborView, SearchResult, SuggestionView, ExplainView, …). The b2 binary parses args, picks the models, calls the façade, and prints — human, or --json for agents.

The invariants that hold it together

Almost every choice above serves one of these. Each is pinned by the suite.

B2 never authors the bodyWrites are surgical and in frontmatter: the b2id stamp, and an accepted edge in relations:. → stamp · frontmatter · accept

Not yet built

An honest architecture names its boundaries. The v1 kernel is feature-complete; what remains is tuning, scale, and packaging — tracked in planning/tasks.md, not gaps that slipped through.

  • Relator-quality tuning (parked). The suggestion-quality eval and a first baseline ship (precision 0.82 / recall 1.00 / verb-acc 0.93 over 22 pairs on claude-opus-4-8, 2026-07-03), but one run isn't enough signal to change the prompt or model. Resume with a variance pass, then grow the labelled set, then tune one lever at a time (specs/eval-strategy.md §6).
  • Semantic quality in CI. The engine suite runs on the deterministic fakes, so it proves plumbing, not ranking quality. The real model is measured by two separate out-of-CI evals — retrieval (b2-embed) and suggestion-quality (b2-relate).
  • Cold-index scale. Incremental reindex is done; the first embed of a large vault still blocks. Deferred approaches — a background reindex + b2 status, a keyword-first progressive index, a smaller/quantized embedder — all compose behind the existing seams.
  • Deferred, behind clean seams. A cross-encoder reranker (changes ordering, not the store), query expansion, the qmd-style ~900-token overlap-aware chunker (a pure re-projection — the chunks schema already carries token_count + heading_path), a durable audit log of model calls, and the actual packaging/distribution build.
