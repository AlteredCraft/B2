Inside b2 suggest

Exactly what happens when you run b2 suggest: how candidate generation reads the index, how the real Claude-backed relator turns each candidate pair into a typed, explained suggestion, and what a run costs. A companion to the architecture's candidate engine and relator seam.

⚠️Not recommended yet
Running b2 suggest on a real vault can make thousands of paid API calls.

Candidate generation is free, but the relator makes one Claude call per candidate pair — roughly notes × --top calls on a first run. A few hundred notes is already thousands of requests (and the matching token spend). We do not recommend running it as it stands; treat this page as a description of how the pipeline works, not an endorsement of running it at scale.

A lower-cost path is in progress — most likely a local-LLM relator backend behind the same Relator seam — so discovery can run without a per-pair API bill. Until then, if you do try it, keep the vault small, use --top 2, and set model = "claude-haiku-4-5". The cost mechanics are detailed in §3 below.

Prerequisite — a reindexed index. suggest reads the vectors + typed graph a prior b2 reindex wrote — it reads stored vectors straight back and never re-embeds, so it needs no model of its own. But if that index was built under B2_EMBEDDER=fake, the vectors are blake3 hashes, not semantics — candidate generation then surfaces meaningless neighbours and the relator judges nonsense. For real suggestions the order is b2 init → b2 reindex (real model) → b2 suggest.

1The pipeline, per candidate

crates/b2-core/src/discover.rs · crates/b2-relate/ · crates/b2-cli/src/main.rs

The command visits every note as an anchor (in sorted b2id order, so the run is reproducible), generates its candidates, and asks the relator to judge each one. Every candidate lands in exactly one of four buckets — the accounting the CLI reports.

generated (written) declined / non-core (pruned) existing (skipped, no call) the model call

generated + declined + non_core + existing = the pairs considered — the tally the CLI prints on stderr. The cost gate is pre-call: an already-settled pair (a pending suggestion or a rejection tombstone) is skipped before the paid relate(), so a re-run pays only for new pairs. Everything downstream of relate() is precision — the relator's decline, the is_core check, and finally you. (generate_suggestion keeps a per-type guard as a backstop.)

2The relator, made real

crates/b2-relate/ — config.rs · prompt.rs · claude.rs

The middle stage — turning a pair into a typed verdict — is the one step that needs a model. It lives behind the same Relator trait the fake uses, in its own crate so b2-core stays model-free (the b2-embed precedent). The shipped backend is Claude over the Messages API.

How one pair is judged
  • Forced tool use. One classify_relation tool whose relation is an enum of the closed core verbs, forced via tool_choice — so the model returns a typed verdict, never free text to repair.
  • Raw HTTP, synchronous. Rust has no official SDK, so it's a plain ureq POST — no tokio pulled into the tree.
  • Still re-validated. The pipeline re-checks is_core — a real model's verb is verified, not trusted.
Config — pluggable, Claude first

A [relator] table in the same global TOML the embedder reads. The backend is selectable (a local one can drop in behind the seam later); Claude is the default. The API key is never in the file — it comes from ANTHROPIC_API_KEY and is validated at startup, so a missing key fails fast with an actionable message, never a mid-run 401.

# ~/.config/b2/config.toml — all optional
[relator]
backend   = "claude"            # the default; the seam for a future local backend
model     = "claude-opus-4-8"   # override to "claude-haiku-4-5" for cheap, high-volume runs
base_url  = "https://api.anthropic.com"
max_tokens = 1024
Offline & deterministic when you need it. B2_RELATOR=fake (or the blanket B2_EMBEDDER=fake) swaps in the content-addressed FakeRelator — no network, no key — which is exactly how the model-free test suite drives suggest. The loud "stub relator" caveat prints only on that path; under the real relator it comes off.
Ground truth: b2-relate — 10 model-free tests (config defaults / overrides / unknown-backend; the request forces the tool, pins the verb enum, and carries the evidence; response parse covers fired / decline / verb-less-degrades / confidence-clamp / no-tool-call-errors) plus an #[ignore]d live smoke test. The CLI's suggest tests drive the whole flow on the fake. 127 workspace tests green.

3What a run costs — and how to run it well

The honest operator view. The relator is a network model, so suggest is the one command whose cost scales with your vault and your token budget. The number that matters is calls — and calls scale with notes × top, not with your chunk count. On the first run; a re-run is incremental (see below). The run surfaces this as it goes: a live judging… i/N call(s) line while it works, then a summary with the actual token total (transient — nothing is written to disk).

A worked example

A vault of 200 notes, ~1,400 chunks (≈7 per note), run at --top 5:

  • Candidate generation is pure vector math over the 1,400-chunk pool — no model, no spend. It yields up to 5 candidates per anchor.
  • The relator makes one Claude call per candidate → up to 5 × 200 = ~1,000 calls on the first run (fewer where a note has under 5 unconnected near-neighbours).
  • A re-run is cheap: pending suggestions and rejected pairs are skipped before the model (pre-call dedup), and accepted pairs drop out entirely. It re-pays only for pairs still without a verdict — chiefly earlier declines, which leave no edge to skip.
  • Levers: --top 2 → ~400 calls; model = "claude-haiku-4-5" → a fraction of the per-call cost. Chunk count sets the KNN pool (candidate-gen compute), not the call count.

4Accept & reject — closing the loop

A suggestion is inert until you act on it — it lives only in the event log and the disposable index, never in your Markdown. Two terminal moves resolve it (the suggestion lifecycle):

accept

Appends the typed link to the source note's frontmatter relations: — Markdown-first, never the body — and re-projects it as an authored edge. Re-projection re-embeds that one note, so accept loads the real model like reindex.

reject

Appends a tombstone event so the same pair + type is never proposed again. Touches no vectors, needs no model — just the log.

The queue survives a rebuild. Drop the index and it rebuilds from Markdown; replay the log and the pending queue, the accepted edges, and the rejection tombstones all come back — the review state is the one thing the log, not the Markdown, is authoritative for.
